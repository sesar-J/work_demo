# 2026-04-01 项目架构设计与落地说明

---

## 1. 项目要解决什么问题

这个项目是一个案例实操中心，目标不是“只看文档”，而是把“看案例 + 在线动手”合并成一条用户路径。

用户路径如下：

1. 首页选案例；
2. 详情页了解案例背景和目标；
3. 操作页左侧看步骤文档；
4. 操作页右侧进入 Notebook 执行命令完成实操。

当前我实现的版本，是可演示、可联调、可扩展的阶段版本。

---

## 2. 已确认的产品约束

我按以下口径实现并设计：

1. 案例内容以 `ipynb` 为唯一源；  
2. 右侧 Notebook 按“用户 + 案例”创建独立会话；  
3. 用户退出再进，不保留历史状态；  
4. Git event 只监听 DTSE 案例内容仓；  
5. Terraform 作为所有案例统一环境能力；  
6. AK/SK 由用户输入。

另外，当前阶段优先“先做出来”，不做 AK/SK 安全治理，不做持久化。

---

## 3. 总体架构（业务视角）

一句话：当前架构是“内容中心 + 会话入口层”。

```text
DTSE内容仓(Git)
   │ webhook
   ▼
FastAPI后端
  ├─ 内容转换层：ipynb -> html
  ├─ 同步层：签名校验、仓过滤、分支过滤、增量重建
  ├─ Lab层：会话URL生成、Terraform模板下发
  └─ SQLite：案例索引、同步日志
   ▲
   │ API
   ▼
Vue3前端
  ├─ 首页
  ├─ 详情页
  └─ 操作页（左文档 + 右iframe）
```

这套架构重点是先把主链路跑通，再迭代“右侧真实文件工作区”能力。

## 4. 当前代码结构和职责分工

## 4.1 前端

主要文件：

- `frontend/src/views/HomeView.vue`：案例列表、同步状态
- `frontend/src/views/CaseDetailView.vue`：案例详情
- `frontend/src/views/CaseLabView.vue`：操作页（左文档、右 Notebook）
- `frontend/src/api.js`：后端接口封装

职责：

- 渲染页面；
- 调用后端 API；
- 不直接访问案例仓文件。

## 4.2 后端

主要文件：

- `backend/app/main.py`：应用入口
- `backend/app/api/routes_cases.py`：案例接口
- `backend/app/api/routes_sync.py`：同步接口
- `backend/app/api/routes_lab.py`：Lab接口
- `backend/app/services/content_service.py`：内容转换
- `backend/app/services/git_sync_service.py`：git 拉取与签名校验
- `backend/app/services/lab_service.py`：会话 URL 生成和模板输出

职责：

- 管理内容索引；
- 执行同步与转换；
- 返回会话 URL；
- 输出 Terraform 模板。

## 4.3 内容目录

每个案例目录结构：

```text
backend/content/cases/<case-dir>/
  case.json
  detail.ipynb
  operation.ipynb
```

转换结果在：

```text
backend/content/generated/<slug>/
  detail.html
  operation.html
```

## 5. 数据库设计（当前状态）

数据库：`backend/app.db`（SQLite）

当前只有两张核心表：

## 5.1 `case`（案例索引表）

字段说明：

- `slug`：案例唯一标识
- `title` / `summary` / `cover_image`：展示信息
- `detail_html_path` / `operation_html_path`：转换后的 HTML 路径
- `notebook_iframe_url`：该案例对应 Notebook 基础地址

设计意图：

- 数据库不存大段正文；
- 正文在文件（HTML），数据库存索引和路径，轻量且清晰。

## 5.2 `synceventlog`（同步日志表）

字段说明：

- `source`：来源（manual / webhook）
- `status`：成功或跳过
- `message`：细节信息（跳过原因、重建数量等）
- `created_at`：时间

设计意图：

- 让同步可观测、可追踪、可排障。

## 6. 后端接口设计（当前可用）

## 6.1 基础接口

- `GET /api/health`：探活

## 6.2 案例接口

- `GET /api/cases`：案例列表
- `GET /api/cases/{slug}`：案例详情（含 `detail_html`、`operation_html`、`notebook_iframe_url`）

## 6.3 同步接口

- `POST /api/sync/git-event`
  - 校验签名；
  - 校验 DTSE 仓；
  - 校验分支；
  - `git pull`；
  - 增量重建；
  - 写日志。

- `POST /api/sync/rebuild`：手工全量重建
- `GET /api/sync/status`：查看最新同步状态

## 6.4 Lab接口

- `POST /api/lab/session`
  - 作用：创建/更新本案例会话 URL
  - 入参：`case_slug`（必填），`ak/sk`（可选且成对）
  - `user_id`可不传，后端自动 guest 标识
  - 出参：`session_id`、`notebook_url`、`expires_at`

- `GET /api/lab/terraform-template`
  - 作用：返回统一 Terraform 文件模板

## 7. 关键疑问：明确回答

## 7.1 右侧 JupyterLab 里“文件”当前到底来自哪里？

当前右侧文件树来自外部 Notebook 平台，不是本后端把案例文件真实挂载进去。

也就是说：

- 左侧文档是我系统转换并可控的；
- 右侧文件目前由 iframe 目标环境决定。

## 7.2 当前嵌入 iframe 的本质是什么？

是“Notebook Web 会话入口 URL”，用于打开实操环境页面。  
它不是“本地目录映射”本身。

## 7.3 每个案例 iframe 是否独立？

当前是“案例可配置 + 会话隔离”：

- 案例级：每个案例可配置不同 `notebook_iframe_url`
- 会话级：每次进入生成新 `session_id`

但是否是“真正独立容器”，取决于 Notebook 平台，不由 URL 参数单独决定。

## 7.4 AK/SK 的作用是什么？当前怎么用？

作用：用户执行 Terraform/云 API 操作时鉴权。  
当前用法：用户输入后作为会话参数更新，不做复杂安全托管和持久化。

## 7.5 “右侧是真实案例文件”是否已实现？

当前**未完全实现**。  
当前实现的是“会话入口独立”，不是“真实案例文件工作区注入”。

1. `ipynb` 唯一源；
2. DTSE 仓同步监听；

4. 操作页会话入口可用；
5. AK/SK 输入流程可用；
6. Terraform 统一模板接口可用。

## 8.2 部分满足

## 8.3 未满足

1. 右侧一进入就出现本案例真实文件树；
2. 会话持久化、回收与调度治理；
3. 运行时审计和稳定性治理。

## 9. 下一步落地方案（按你当前口径：先简单可用）

你已经明确“先做简单，不做复杂安全，不做持久化”，则建议如下：

## 9.1 MVP目标

右侧打开即进入真实 Jupyter，会看到本案例文件，退出重进恢复初始状态。

## 9.2 实施步骤

1. 进入操作页时后端创建临时 `session_id`；
2. 在临时目录写入案例文件（含统一 Terraform 模板）；
3. 启动/分配真实 Jupyter 运行时并挂载该目录；
4. 返回会话 URL 给前端 iframe；
5. 会话过期或页面关闭后清理临时目录；
6. 不做数据库持久化（符合你当前阶段要求）。

## 9.3 对当前系统影响

- 后端：增强 `lab_service` 与 `routes_lab`
- 前端：保留现有操作页，仅强化“自动会话 + 重置环境”体验
- 数据库：当前阶段可不新增表

当前系统已经完成“内容中心 + 会话入口”的核心演示能力。
