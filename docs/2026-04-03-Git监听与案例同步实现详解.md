# Git 监听与案例同步实现详解（面向项目新人）

本文面向第一次接触本项目的人，目标是回答三个问题：

1. 系统是如何“监听 Git 变更”的？
2. Git 变更后，案例内容如何自动更新到页面？
3. 目前实现到了哪一步、有什么边界和后续优化方向？

---

## 1. 先理解业务目标

这个项目的案例内容由 DTSE 仓维护。我们希望做到：

- DTSE 在仓库更新案例（主要是 `ipynb`）后，平台自动感知变化。
- 平台自动拉取最新内容，并把 notebook 转成网页可展示内容。
- 首页、详情页、操作页在下一次请求时看到最新案例内容。

这就是“Git 监听 + 内容同步”的核心目标。

---

## 2. 一句话架构

当前实现采用 **Webhook 驱动的后端同步**：

- Git 平台（CodeArts / GitCode）推送事件 -> 回调项目后端接口。
- 后端验证签名 -> 过滤仓库与分支 -> `git pull` -> 增量重建案例。
- 重建时把 `ipynb` 转成 HTML，更新数据库索引。
- 前端再通过 API 读取更新后的 HTML 内容。

---

## 3. 代码位置总览

核心代码在这些文件：

- `backend/app/api/routes_sync.py`：Webhook 接口、事件过滤、同步主流程、同步状态接口
- `backend/app/services/git_sync_service.py`：签名校验、执行 `git pull`
- `backend/app/services/content_service.py`：解析案例、`ipynb -> HTML`、增量重建
- `backend/app/api/routes_cases.py`：给前端返回案例列表和详情（含 HTML）
- `backend/app/models.py`：`Case` 与 `SyncEventLog` 数据模型
- `backend/app/config.py`：内容目录与数据库路径

---

## 4. 详细执行链路（从 push 到页面更新）

### 第 1 步：Git 平台推送事件到后端

DTSE 仓发生 push 后，Webhook 会调用：

- `POST /api/sync/git-event`

接口入口在 `routes_sync.py` 的 `sync_from_git_event()`。

### 第 2 步：安全校验（签名或 token）

后端取出请求体与签名头，调用：

- `verify_webhook_signature(...)`

校验逻辑在 `git_sync_service.py`：

- 如果配置了 `GIT_WEBHOOK_SECRET`，就必须验证通过。
- 支持 GitCode token 校验（`x-gitcode-token`）。
- 校验失败返回 401，流程终止。

### 第 3 步：仓库与分支过滤

避免“别的仓、别的分支”触发同步，后端做两层过滤：

- 仓库过滤：`_is_dtse_repo(payload)`
  - 通过环境变量 `DTSE_REPO` 精确匹配（推荐）
  - 未配置时使用包含 `dtse` 的兜底判断
- 分支过滤：`_extract_branch(payload)` + `SYNC_BRANCHES`
  - 默认允许 `main,master`

若不满足条件，接口会记录一条 `skipped` 日志并返回“已跳过”。

### 第 4 步：提取变更路径（用于增量）

后端从 payload 中收集改动文件路径：

- `commits[*].added/modified/removed`
- `head_commit.added/modified/removed`
- `push_data.commits[*]...`（兼容不同平台 payload）

提取方法：`_extract_changed_paths(payload)`。

### 第 5 步：拉取内容仓最新代码

调用 `pull_latest_content()` 执行：

- `git -C backend/content pull`

说明：

- 当前设计默认 `backend/content` 是一个已经初始化好的 Git 仓（DTSE 内容仓本地副本）。

### 第 6 步：重建案例（核心）

调用：

- `rebuild_cases_from_ipynb(session, changed_paths=...)`

重建逻辑在 `content_service.py`，关键动作如下：

1. 扫描 `backend/content/cases/*/case.json`
2. 读取每个案例清单：
   - `slug`
   - `detail_notebook`
   - `operation_notebook`
3. 若有 `changed_paths`，先推断候选案例目录，只重建受影响案例（增量）
4. 用 `nbconvert.HTMLExporter` 将 ipynb 转为 HTML：
   - `detail.ipynb -> content/generated/<slug>/detail.html`
   - `operation.ipynb -> content/generated/<slug>/operation.html`
5. 写回数据库 `Case` 表（更新标题、摘要、封面、HTML 路径、iframe 地址）

### 第 7 步：记录同步日志

每次同步（成功/跳过）会写入 `SyncEventLog`：

- `source`（manual / codearts-or-gitcode）
- `status`（success / skipped）
- `message`（保留技术细节，便于排障）

### 第 8 步：前端读取最新内容

前端访问：

- `GET /api/cases`：案例列表
- `GET /api/cases/{slug}`：案例详情

后端从 `Case` 表找到 HTML 文件路径，再读文件返回：

- `detail_html`
- `operation_html`

因此同步完成后，页面会在下次拉取数据时显示新内容。

---

## 5. 数据模型怎么配合这条链路

### `Case` 表（案例索引）

作用：把“案例元数据 + 渲染产物路径”组织成前端可直接消费的数据。

关键字段：

- `slug`：案例唯一标识
- `title` / `summary` / `cover_image`
- `detail_html_path`：详情 HTML 文件路径
- `operation_html_path`：操作页左侧 HTML 文件路径
- `notebook_iframe_url`：右侧 iframe 基础地址

### `SyncEventLog` 表（同步审计）

作用：记录同步链路是否成功、为何跳过、技术输出是什么。

---

## 6. 手动同步与状态查看

除了 webhook 自动触发，还有两个运维接口：

- `POST /api/sync/rebuild`：手动全量重建
- `GET /api/sync/status`：查看最新一次同步状态

其中状态接口会把内部技术字符串转换成面向用户的友好文案，页面展示更易理解。

---

## 7. 当前实现边界（必须知道）

### 已实现

- Webhook 驱动监听
- 签名/Token 校验
- DTSE 仓与分支过滤
- `ipynb -> HTML` 自动转换
- 增量重建与日志记录
- 前端友好同步状态展示

### 未实现或待增强

- 失败重试队列（当前失败即返回错误）
- 更强的幂等与并发保护（高频 webhook 场景）
- 精细化告警（如飞书/邮件告警）
- 多仓多租户的隔离策略

---

## 8. 未来与“右侧真实 TF 文件”的关系

当前同步链路主要管理左侧文档内容（`ipynb -> HTML`）。  
如果未来要让右侧 JupyterLab 里出现真实 `.tf` 文件，建议扩展为：

- DTSE 仓新增 `cases/<case>/lab-files/*.tf`
- 会话启动时把 `lab-files/` 注入云开发会话目录
- 右侧 iframe 打开该会话目录

这样可以实现“左侧文档 + 右侧真实文件”同源，减少教学与实操不一致问题。

---

## 9. 运维配置清单（上线前最少项）

- `GIT_WEBHOOK_SECRET`：Webhook 安全密钥
- `DTSE_REPO`：允许同步的 DTSE 仓标识
- `SYNC_BRANCHES`：允许触发同步的分支列表（默认 `main,master`）
- Webhook 地址：`POST /api/sync/git-event`

---

## 10. 给新人一句话理解

本项目的 Git 监听不是“主动轮询 Git”，而是“被动接收 Webhook”。  
收到事件后，后端做校验与过滤，再把 DTSE 仓的 notebook 转成 HTML 并更新数据库索引，前端读取索引展示最新案例内容。
