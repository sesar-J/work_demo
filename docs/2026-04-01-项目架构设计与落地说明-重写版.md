# 2026-04-01 项目架构设计与落地说明

---

## 一、项目目标与当前阶段定义

本项目是一个 Hands-on 实践中心，目标是把“案例学习”和“在线实操”融合成一条连续流程：

1. 用户在首页选择案例；
2. 在详情页理解背景与目标；
3. 在操作页左侧看步骤文档；
4. 在操作页右侧进入 Notebook 执行命令完成实操。

我当前把项目定位为“可演示、可联调、可扩展”的阶段版本
核心是先打通主链路，再逐步增强右侧真实实操能力。

---

## 二、已确认的产品口径（实现依据）

我按以下口径推进了当前实现：

1. 案例内容唯一源是 `ipynb`；
2. 右侧 Notebook 按“用户 + 案例”创建独立会话；
3. 用户退出重进后不保留历史状态；
4. Git event 只监听 DTSE 案例内容仓；
5. Terraform 是所有案例统一实践能力；
6. AK/SK 由用户输入；
7. 当前阶段优先快速落地，不做复杂安全托管和会话持久化。

---

## 三、当前架构设计

## 3.1 业务层架构

系统分为两层：

1. **案例内容中心**
   - 管理案例目录；
   - 将 `ipynb` 转换为前端可展示 HTML；
   - 处理 DTSE 仓内容同步。

2. **实操会话入口层**
   - 为操作页生成会话 URL；
   - 承载用户 AK/SK 输入；
   - 打开右侧 Notebook iframe。

## 3.2 技术实现架构

```text
DTSE案例仓(Git)
    │ webhook
    ▼
FastAPI后端
  ├─ routes_cases: 案例查询
  ├─ routes_sync: 同步控制
  ├─ routes_lab: 会话与模板
  ├─ content_service: ipynb->html
  ├─ git_sync_service: 校验+拉取
  ├─ lab_service: 会话URL生成
  └─ SQLite: case + synceventlog
    ▲
    │ REST API
    ▼
Vue3前端
  ├─ HomeView
  ├─ CaseDetailView
  └─ CaseLabView(左文档+右iframe)
```

## 四、后端数据库设计（重点说明）

数据库文件：`backend/app.db`（SQLite）

当前数据库设计遵循“轻索引、强文件”的原则：正文在文件，数据库存索引和状态。

## 4.1 `case` 表（案例索引）

核心字段：

- `slug`：案例唯一标识；
- `title` / `summary` / `cover_image`：展示元数据；
- `detail_html_path` / `operation_html_path`：转换后的内容路径；
- `notebook_iframe_url`：该案例默认 Notebook 基础地址。

设计目的：

- 快速支持首页列表、详情页查询；
- 不把大段正文塞进数据库，降低维护复杂度；
- 保持内容仓与展示层解耦。

## 4.2 `synceventlog` 表（同步审计）

核心字段：

- `source`：触发来源（manual/webhook）；
- `status`：执行状态（success/skipped）；
- `message`：执行详情；
- `created_at`：时间。

设计目的：

- 让同步过程可追踪；
- 出现“内容没更新”时可快速定位原因。

## 五、后端接口设计（重点说明）

## 5.1 基础接口

- `GET /api/health`
  - 作用：探活。

## 5.2 案例接口

- `GET /api/cases`
  - 作用：首页案例列表。

- `GET /api/cases/{slug}`
  - 作用：详情页与操作页数据；
  - 返回：`detail_html`、`operation_html`、`notebook_iframe_url` 等。

## 5.3 同步接口

- `POST /api/sync/git-event`
  - 作用：接收 DTSE 仓 webhook；
  - 处理：签名校验 -> 仓过滤 -> 分支过滤 -> `git pull` -> 增量重建 -> 写日志。

- `POST /api/sync/rebuild`
  - 作用：手动全量重建内容。

- `GET /api/sync/status`
  - 作用：查看最近同步状态。

## 5.4 Lab 接口

- `POST /api/lab/session`
  - 作用：创建/更新操作页独立会话 URL；
  - 入参：`case_slug`（必填），`ak/sk`（可选且成对）；
  - 出参：`session_id`、`notebook_url`、`expires_at`。

- `GET /api/lab/terraform-template`
  - 作用：下发统一 Terraform 模板内容。

## 六、关键疑问

## 6.1每个案例 iframe 是否独立？

当前是“案例可配置 + 会话隔离”：

- 案例级可配置不同 `notebook_iframe_url`；
- 会话级每次创建新 `session_id`。

但“是否真正独立容器”取决于目标 Notebook 平台。

## 6.2 AK/SK 后续 ？

AK/SK 用于用户执行 Terraform/云 API 的鉴权语义。  
当前系统做的是输入与会话参数透传，不做复杂托管和持久化。

## 6.5 “右侧真实案例文件”是否已实现？

当前未完全实现。  
实现的是“会话入口独立”，未实现“真实案例文件工作区注入”。
